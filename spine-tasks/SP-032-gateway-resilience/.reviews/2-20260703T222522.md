# Code Review — SP-032 Step 2: Testing and verification

**Review Level:** 3 (Full — Plan + Code + Test)

---

### Verdict: REVISE

### Summary

The circuit breaker, weighted failover, and rate limiting implementations are architecturally sound and well-structured. Typecheck passes, all 554 tests pass. However, at Review Level 3 two test-depth issues are blocking: (1) the failover integration test has a tautological assertion that doesn't verify failover actually activates, and (2) the rate limiter has an untested edge case where `allowed: false` with `retryAfterSeconds: null` silently allows the request through — this needs either a fix or a test documenting the intent.

### Blocking issues

1. **`tests/unit/gateway-dispatch.test.ts:97-115`** — The failover test assertion is tautological. The condition `if (decision.selected_model_id === 'econ-a') { expect(...).not.toBe('circuit_breaker_failover'); }` can never meaningfully fail: if the circuit breaker works, the result is `econ-b` (condition false, body skipped); if it doesn't work, the result is `econ-a` with reason `safe_cloud_default` (condition true, assertion trivially passes). The test should deterministically assert that failover fires:
   ```ts
   expect(decision.selected_model_id).toBe('econ-b');
   expect(decision.reason_code).toBe('circuit_breaker_failover');
   ```

2. **`src/infrastructure/gateway/gateway-dispatch.ts:130`** — `dispatchWithRateLimit` guards the 429 response with `!bucket.allowed && bucket.retryAfterSeconds !== null`. When `allowed: false` and `retryAfterSeconds: null`, the request is silently dispatched instead of being rejected. If this is intentional fail-open behavior, add a test in `tests/unit/gateway-dispatch.test.ts` that explicitly asserts a request goes through when the limiter returns `{ allowed: false, retryAfterSeconds: null }`. If it's a bug, the guard should be `if (!bucket.allowed)` with a fallback `retry_after_seconds: 0` when the value is null.
   **Missing test:** `tests/unit/gateway-dispatch.test.ts` — add case: "routes request when limiter denies but retryAfterSeconds is null" (or fix the guard and test the 429 response).

### Suggestions (non-blocking)

- **`tests/unit/gateway-dispatch.test.ts:207-237`** — `weightedSelect` tests verify it returns a candidate but don't test the weighting distribution. Consider a statistical test (run N=1000 iterations, assert the lower-cost model gets ≥60% of selections) to validate the inverse-cost weighting logic.

- **`src/infrastructure/gateway/circuit-breaker.ts:59`** — `statusCode === 429` is treated as infra error for the circuit breaker, but the gateway's own rate limiter also returns 429. If an upstream 429 trips the breaker and the local limiter also fires, the model could be doubly penalized. Consider documenting this interaction or separating upstream-429 from local-rate-limit in the `isInfraError` classification.
